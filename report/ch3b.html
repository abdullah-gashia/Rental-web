3.4 System Implementation

3.4.1 Frontend Implementation

For the frontend I went with Next.js 14 because it's what I'm most comfortable with, and the App Router makes routing pretty straightforward once you understand the folder-based convention. Every folder under app/ becomes a route automatically, and you can co-locate server components, client components, and even API routes in the same directory, which keeps things organized.

The folder structure is organized roughly as follows:

  • app/ — Pages using the App Router. Key routes include / (homepage marketplace), /rental/orders/[id] (order detail), /dashboard/rentals (user's rental dashboard), /admin/* (admin panel), and /rental/orders/[id]/receipt (printable PDF receipt).
  • components/ — Reusable React components, organized by category: components/items/ for product cards and detail views, components/rental-checkout/ for the checkout wizard, components/rental/ for handshake components, components/layout/ for navigation and layout.
  • lib/ — Server actions, Prisma client, type definitions, Zustand stores, hooks, and utility functions.
  • prisma/ — Prisma schema file and migration history.

Tailwind CSS handles all the styling. I used a custom design system built around a warm off-white background (#faf9f7), an orange primary action color (#e8500a), and a near-black for text. The design is clean and minimal — this is an internal university tool, not a consumer app that needs to be visually flashy.

Client-side state is managed through Zustand stores. There are separate stores for: authentication state (current user, role, session), locale (Thai/English i18n), notifications, wishlist count, and toast messages. Using Zustand rather than prop-drilling or React Context everywhere keeps the components clean and makes it easy to access global state from anywhere without wrapper hell.

Internationalization (i18n) is handled through a simple dictionary system in lib/i18n/. The dictionaries define Thai and English translations for all UI strings, and a locale store lets users toggle between languages. The platform defaults to Thai since the target users are primarily Thai students.

3.4.2 Backend Implementation

The backend uses Next.js Server Actions almost exclusively for data mutations. Server actions are async functions marked with "use server" that run on the server but can be called directly from client components. This is actually a pretty clean pattern because it removes the need to write separate REST endpoints for most operations — the function is the API.

For reads, most pages are server components that call database functions directly (since they run on the server anyway), and for any reads that need to happen client-side after interaction, the server actions return data directly.

3.4.3 Authentication and Authorization

Authentication is handled by NextAuth.js. Users can sign up with email and password, or sign in with Google OAuth. NextAuth manages session tokens and provides the current user object to both server and client code via the session callback.

Role-based access control has three levels: RENTER (default), OWNER (same user, gains owner capabilities when they list an item), and ADMIN. In practice, RENTER and OWNER aren't separate roles — the same user can do both. The distinction is just that certain actions (like approving requests or signing return receipts as the owner) check whether you are the ownerId on the specific order.

ADMIN is a hard role assigned in the database. Admin users see the admin panel link in navigation and can access /admin/* routes. On the product detail page, admins see a read-only "Admin View" badge instead of buy/rent buttons — this prevents admins from accidentally creating orders while monitoring the platform.

Banned users (where isBanned = true in the database) are rejected at the NextAuth signIn callback before the session is created, so they can't log in at all. KYC verification is a gate specifically for listing items — an unverified user can browse and view items, but the server action for creating a listing checks verificationStatus === "APPROVED" and returns an error if not met.

3.4.4 The Rental Flow (End-to-End)

This section walks through the complete rental journey from the perspective of both parties, explaining what happens in the UI and what happens in the backend at each step.

A. Listing an Item

An owner who wants to rent out their camera, for example, clicks the "ลงประกาศ" (Post Ad) button in the navigation. A modal opens with a three-step form. In step one they fill in the title, category, condition, and description. Step two handles photos — they can upload multiple images and designate one as the main image. Step three is where the rental configuration happens: they set the rental rate type (daily, monthly, or yearly), the rate amount, a security deposit amount, minimum and maximum rental duration, late fee per day, and any specific rental terms or instructions.

When they submit, a server action validates the data, checks that the user is KYC-verified, creates the Item record with status: "PENDING", saves the images to storage, and creates ItemImage records. The item goes into an admin review queue and doesn't appear on the marketplace until an admin approves it.

C. Owner Approval

When the checkout is submitted, the system creates a RentalOrder record with status REQUESTED, generates a unique reference code, stores the agreed rental terms as a JSON text snapshot (so any future changes to the item don't affect the agreed terms), and sends a notification to the owner.

The owner sees the request in their rental dashboard and on the order detail page. They have 24 hours to respond before the request automatically expires. If they approve, a Prisma transaction debits the renter's wallet (rental fee + platform fee + security deposit), records the wallet transaction, updates the order status to APPROVED (then immediately auto-transitions to DEPOSIT_HELD), and notifies the renter.

D. Handover — Digital Handshake #1

Once the deposit is held, both parties can access the Digital Handshake panel on the order detail page. The panel shows two status indicators: one for the renter's confirmation and one for the owner's confirmation. Both need to confirm for the handshake to complete.

Each party uploads photos of the item's current condition (this is mandatory — the upload button is prominent and the confirm button stays disabled until at least one photo is uploaded). The renter, specifically, also needs to complete a digital signature. The signature component shows the full rental agreement text above a canvas element where they draw their signature with a mouse or finger. When they confirm the signature, the canvas data URL is captured and stored.

E. Active Rental Period

During the ACTIVE state, the order detail page shows a countdown to the return deadline. If the return date is within 24 hours, the banner turns amber. If it has passed, it turns red with an overdue warning. The cron job runs once per day and checks all active orders. Any order where rentalEndDate < now is automatically transitioned to OVERDUE status, and a late fee is added to the order daily.

If the item was listed as renewable (isRenewable: true), the renter can request an extension from the order page. This sends a request to the owner, who approves or rejects. If approved, the end date is extended and the additional fees are calculated and charged.

F. Return — Digital Handshake #2

When the renter is ready to return the item, they click "นัดคืนของ" on the order page. This schedules a return meetup (same campus location picker as pickup) and moves the order to RETURN_SCHEDULED.

At the return meetup, the return handshake panel becomes active. This works differently from the pickup handshake: both parties still upload photos, but this time it's the owner who does the condition assessment. The owner sees a condition selector (Same / Minor Damage / Major Damage / Lost) and if damage is found, an input for the damage fee amount. The owner then signs a digital return receipt using the same signature canvas. Once both parties confirm, the order moves to RETURNED.

G. Settlement

The settlement calculation runs automatically when the order reaches RETURNED status. The logic is:

  • Rental fee (already held in escrow) is released to the owner, minus the 5% platform fee.
  • Late fees (if any accumulated during OVERDUE periods) are added to the owner's payout.
  • Damage fee (if the owner set one at return) is deducted from the renter's deposit refund.
  • Remaining deposit is refunded to the renter's wallet.
  • If the condition is LOST, the full deposit is forfeited to the owner.

The status moves to COMPLETED or COMPLETED_WITH_DEDUCTION depending on whether any deductions were made. All wallet movements are recorded as individual WalletTransaction entries with a descriptive type field.

H. PDF Receipts

After the order reaches ACTIVE status, both parties can access the rental contract document at /rental/orders/[id]/receipt?type=contract. After COMPLETED, the return receipt is also available at ?type=return. Both pages are print-optimized HTML with @media print CSS that hides the browser navigation and the "Print / Save PDF" button, producing a clean A4 document. The page auto-triggers window.print() on load after a short delay to allow fonts and content to render.

I. Dispute Flow

Either party can open a dispute from the order page if they disagree with the other party's assessment. Opening a dispute freezes the escrow and notifies the admin. The admin can then access a full evidence review page showing all photos from both handshakes, the signatures, the status history timeline, and any messages. Based on this, the admin makes a ruling and manually triggers the appropriate settlement calculation. The dispute mechanism is designed to be a last resort — the Digital Handshake evidence usually makes disputes straightforward to resolve.
