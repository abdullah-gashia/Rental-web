Abstract

This project is about building an Equipment Rental System — basically a web platform where students and staff at Prince of Songkla University, Phuket Campus can list, discover, and rent equipment from each other. The idea came from a real problem I kept seeing around campus: students regularly need cameras, calculators, lab tools, sports gear, or musical instruments for just a few days, but buying these things is way too expensive, and the informal "just ask a friend" method is unreliable and leaves no record of what happened if something goes wrong.

The core problem with the existing approach is accountability. When people lend things through Facebook chat or LINE, there is no deposit system, no condition record, no formal agreement, and no clear process when items come back damaged or late. This project tries to solve that by creating a proper digital infrastructure for peer-to-peer rental within the university community.

The system covers the full rental lifecycle, from posting an item and setting a deposit amount, all the way through to condition inspection, return confirmation, and automatic settlement. Both parties go through KYC verification using their PSU student ID before they can list or rent, so the platform stays limited to the campus community. The deposit is held in an escrow wallet managed by the platform, which means neither party ever holds the other's money directly. Handover and return are both handled through a "Digital Handshake" — each party uploads photos and signs a digital agreement on their device, creating a timestamped evidence record that admins can review if a dispute comes up.

The technical stack is Next.js 14 with the App Router, TypeScript, Tailwind CSS for the frontend, and PostgreSQL with Prisma ORM on the backend. Authentication uses NextAuth.js. The platform currently runs on a local development environment, with Vercel deployment planned for Project II.

At the current stage (Project I), the core rental flow is working end-to-end in a prototype. A small group of five students tested the system and generally found the checkout process clear and the handshake concept easy to understand. A few things still need refinement, especially around mobile touch input for signatures, but the foundation is solid.

Chapter 1: Introduction

1.1 Overview

During my time at PSU Phuket, I noticed a pretty consistent pattern: students regularly need equipment they don't actually own. A third-year student needs a DSLR camera for a media project but only for one weekend. A first-year engineering student needs a scientific calculator for exam week. A group putting on a campus event needs costumes, props, or audio equipment for just two days. For most students, buying these things outright makes no sense financially — it's expensive, and you end up with something you'll use once and then forget about.

The natural solution, at least the way it works now, is to borrow from friends. And that works fine until it doesn't. Items come back late. Items come back damaged with no clear record of when the damage happened. Sometimes items don't come back at all. There's no formal process, no deposit system, and no record anyone can refer to if something goes wrong. Most people just absorb the loss because confronting a classmate over a borrowed item feels awkward, and there's no evidence to point to anyway.

Some students have tried using Facebook groups or LINE group chats to do informal rentals or sales. These work better than nothing, but they have real problems. There's no verification that the person you're dealing with is actually a PSU student. There's no deposit mechanism, so there's no financial accountability. Condition isn't recorded before handover, so disputes over damage are purely word-against-word. And because the transactions happen in chat, there's no structured record — finding an old conversation about a rental from three months ago is basically impossible.

The Equipment Rental System this project proposes is an attempt to fix these problems with a proper digital platform. The basic idea is a web application exclusively for the PSU community where students and staff can list their equipment for rent and rent equipment from others, with the platform managing the deposit, the agreement, and the evidence record. Because both parties verify their PSU identity before using the system (through a KYC process that checks their student or staff ID), the community stays closed and trust is built in from the start.

What makes this different from just posting in a Facebook group is the escrow system. When a renter books an item, they pay a deposit and the rental fee upfront, but that money goes into the platform's escrow wallet — not directly to the owner. The owner only receives their payout after the rental is completed successfully. This gives both sides a reason to follow through properly: the renter doesn't lose their deposit for no reason, and the owner knows the payment is secured. If the item comes back damaged, the platform deducts the damage fee from the deposit based on what both parties agreed when signing the rental agreement.

The "Digital Handshake" is the other core mechanism. At both handover and return, both the owner and the renter independently open the platform on their devices, upload photos of the item, and sign a digital agreement using a canvas-based signature. The platform records the photos, signatures, and timestamps. This creates a tamper-evident evidence record. If someone claims the item was already damaged when it was returned, the photos from the handover and return are both on file and an admin can compare them directly.

Admin plays a specific role in this system as a neutral intermediary. They don't just moderate content — they hold the escrow, they enforce the rules, and in a dispute they function like a small-claims mediator. The admin can see every photo, every signature, every timestamp, and every message that's been exchanged. This "evidence vault" approach is what makes the system viable for handling real disputes between real people who have to see each other on campus.

From a practical standpoint, a system like this could save students real money. A DSLR camera rental for a weekend at ฿300 is much more accessible than buying one for ฿30,000. A costume rental at ฿150 for an event is far cheaper than buying something you'll wear once. Over time, a well-maintained rental platform creates a campus circular economy where people with equipment they don't constantly need can earn a small income from it, and people who need it temporarily can access it affordably.

  1. Requirement Gathering. I surveyed around 30 PSU students from different faculties about their equipment needs, current borrowing habits, and what features they'd want in a rental platform. Chapter 2: Literature Review

2.1 Peer-to-Peer Rental Platforms

The concept of peer-to-peer (P2P) rental — where individuals rent their own assets to other individuals, with a platform acting as the connector and trust facilitator — became mainstream largely through companies like Airbnb and Uber. Airbnb demonstrated that strangers would pay real money to stay in another person's home, given enough trust signals and a reliable payment and review system [1]. This insight eventually spread to physical equipment rental through platforms like Fat Lama (UK-based equipment rental marketplace), RentMy, and similar services.

What these platforms have in common is a set of mechanisms designed to build trust between strangers. Hamari, Sjöklint, and Ukkonen [2] studied motivation factors in the sharing economy and found that both economic benefits and social factors (reputation, community) drive participation. Platforms that invest in reputation systems — verified identity, star ratings, review histories — consistently see higher transaction rates and lower dispute frequencies than those that don't.

For equipment specifically, the main challenges are around condition disputes and late returns. A study by Möhlmann [3] on P2P sharing platform satisfaction identified "item condition matching expectations" as one of the top predictors of whether a user would repeat their experience. When a renter receives an item in worse condition than advertised, or an owner gets an item back damaged, the lack of a documented pre-rental condition record makes resolution nearly impossible without a formal evidence system.

Fat Lama, one of the more established equipment rental platforms, addresses this through a mandatory photo-upload process at both handover and return, combined with an insurance product that covers damage. For a university-scale platform, a full insurance product isn't realistic, but the photo-upload handshake is directly applicable and is what this project adopted.

One characteristic of commercial P2P platforms that doesn't translate well to a university context is their geographic scope. Fat Lama and similar services operate in large cities where strangers are unlikely to have any prior relationship. In a university, most users will know each other or at least recognize each other, which changes the trust dynamic significantly. This means a university rental platform can rely more on institutional identity (student ID verification) and less on star ratings or elaborate reputation scores. The design of this system reflects that — KYC is the primary trust gate, and ratings are a secondary feature rather than the core mechanism.

2.2 Escrow Systems in Digital Transactions

An escrow system, in its basic form, is a financial arrangement where a neutral third party holds money on behalf of two transacting parties until specific conditions are met. In traditional real estate transactions, an escrow company holds the purchase price until title transfer is complete. In digital platforms, the role of escrow agent is played by the platform itself.

Pavlou and Fygenson [4] studied trust mechanisms in e-commerce and identified the escrow model as one of the most effective ways to reduce perceived transaction risk for both buyers and sellers. The key insight is that escrow removes the need for either party to trust the other completely — they only need to trust the platform, which is a lower bar, especially in a transaction involving strangers.

For rental specifically, a deposit-based escrow model is standard in real-world property rental: landlords routinely hold security deposits that are released minus deductions at the end of a tenancy. Translating this to short-term equipment rental, the deposit serves the same purpose — it provides the owner with financial recourse if the item is damaged, while the renter retains the incentive to behave responsibly because the deposit is refundable on good behavior.

From a legal standpoint in Thailand, rental contracts (สัญญาเช่าทรัพย์) are governed by the Civil and Commercial Code, Sections 537 through 571. Section 537 defines rental as an agreement where one party agrees to give another the temporary use or enjoyment of a property in exchange for remuneration. Section 549 establishes that the lessee (renter) must care for the property and is liable for damage resulting from negligence. Section 562 deals with the return of property in its original condition, accounting for fair wear and tear. These provisions give legal grounding to the deposit and damage assessment system this platform implements [5].

One design decision in this project that's worth explaining is why the platform uses a deposit-plus-rental-fee escrow model rather than a pure deposit model. The reason is that collecting both amounts upfront protects the owner from non-payment scenarios — a renter can't pick up an item and then claim they can't pay. The rental fee is held by the platform and paid out to the owner only after return is confirmed, giving the owner an incentive to participate in the return process properly too.

The escrow wallet in this system is implemented as a simple credit ledger: each user has a wallet balance that can be topped up, and transactions are atomic database operations that debit and credit the relevant accounts simultaneously. This is a simplified model — a production system would integrate with a payment gateway and hold funds in a real escrow bank account — but it is sufficient to demonstrate the correct business logic.

2.3 University Marketplace Platforms

Several universities have attempted to build or encourage student marketplace platforms. MIT's SWAP (Student Wallet and Property) and Harvard's Dorm Room Fund have explored P2P exchange in academic settings, though most of these focus on buying and selling rather than rental. A more directly relevant example is the "UniShare" concept explored in several European universities, where students rent textbooks, equipment, and study materials to each other through a university-managed portal [6].

The consistent finding across these initiatives is that geographic concentration and identity verification are significant advantages. When all users are on the same campus, meetups are easy to arrange, the social cost of bad behavior is higher (reputation spreads quickly in a closed community), and the university can serve as a credible identity guarantor. Dinev and Hart [7] found that users are significantly more willing to share personal information and engage in transactions on platforms that are affiliated with a trusted institution — in this case, the university itself.

The main challenge with university platforms, as documented by Benkler [8] in the context of peer production and resource sharing, is critical mass. A rental platform with five items listed is useless. The initial seeding problem — getting enough listings to make the platform useful before the platform is popular enough to attract listings — is a genuine business challenge. For this project, the scope is limited to Phuket Campus specifically (around 5,000 students), which is small enough that even modest adoption (5-10% of students listing items) would produce a meaningful catalog.

One design feature borrowed from successful university platforms is the campus location system. Rather than requiring users to enter custom addresses, the system offers a set of well-known campus meetup spots (the library, the main cafeteria, the engineering building entrance). This reduces friction in the pickup scheduling step and keeps meetups on-campus, which adds a layer of personal safety to transactions.

Chapter 3: Proposed Solution

3.1 Software Specification

The Equipment Rental System is a web-based application that runs in any modern browser. It's designed to be responsive, so it works reasonably well on a phone's browser even though a dedicated mobile app is not part of Project I. The system has three distinct user roles — Renter, Owner, and Admin — each with different views, permissions, and available actions.

The system is organized into the following functional modules:

  • User Management: Registration, login, profile management, KYC submission and review, role assignment, account suspension, and wallet top-up.
  • Listing Management: Equipment posting with photo upload, rental rate configuration (daily/monthly/yearly), deposit setting, availability management, and soft-delete with a 24-hour grace period.
  • Discovery and Search: Homepage browse with AI-based recommendations, category filtering, keyword search, and item detail view with image gallery.
  • Rental Booking (Checkout Wizard): Four-step guided flow covering date selection with pricing preview, meetup scheduling, payment confirmation with wallet balance check, and digital agreement review and acceptance.
  • Payment and Escrow: Wallet system for holding rental fees and deposits, automated settlement calculations, late fee accumulation, and payout tracking.
  • Handshake System: Dual-party photo upload and digital signature capture for both pickup and return events, with condition rating and damage fee setting at return.
  • Order Management: Rental order detail pages, status history timeline, renewal requests, return initiation, and PDF document downloads.
  • Admin Panel: KYC review queue, item moderation dashboard, rental order monitoring, dispute review, user management, and platform statistics.
  • Notification System: In-app notifications for status changes, approaching due dates, overdue alerts, and system messages.

Presentation Layer: The UI is built with Next.js React components using Tailwind CSS. 3.3.3 Admin Use Cases and the Intermediary Role

[PLACEHOLDER FOR FIGURE 3-4: Use Case Diagram — Admin

UML Use Case diagram with Admin actor connected to: View/Manage Users — Approve/Reject KYC — Moderate Item Listings — Monitor Rental Orders — Intervene in Disputes — Manage Escrow Oversight — Generate Platform Reports — Handle Account Deletion — Curate Trending Items]

The admin role is more than just content moderation. In this system, the admin acts as the neutral intermediary (คนกลาง) in every rental transaction, even when things are going smoothly. This intermediary role has four dimensions:

The Evidence Vault. Every action in the rental lifecycle creates a timestamped record: photos uploaded at pickup, photos uploaded at return, digital signatures with timestamps, chat messages, status change history. The admin can access all of this at any point. If a dispute opens, the admin doesn't have to rely on either party's claims — they can look at the actual photos and compare the item's condition at pickup versus return. They can check the signatures to confirm both parties acknowledged the agreement. This evidence-first approach is what makes the dispute resolution process fair rather than arbitrary.

The Escrow Agent. The admin (representing the platform) holds the rental fee and security deposit from the time the request is approved until the rental is completed. Neither the renter nor the owner ever holds the other's money directly. This means a renter can't claim to have paid when they haven't, and an owner can't pocket the deposit unfairly. All financial movements happen through the platform's wallet system, which keeps a full transaction history. Even the admin cannot arbitrarily move funds — the settlement is calculated by the system according to the rules both parties agreed to at booking.

The Tracker. The system automatically monitors rental periods and flags overdue orders. The cron job runs daily and identifies any orders where the rentalEndDate has passed without the renter initiating a return. It applies a configurable daily late fee to the escrow amount and sends notifications to both parties. This removes the awkward situation where an owner has to personally chase a renter for a late return — the system does it automatically and creates a financial incentive to return on time.

The Court. When a dispute is opened, the admin reviews the complete evidence record and makes a ruling on how the escrow should be distributed. They can access both parties' uploaded photos, the digital signatures, the agreed-upon damage terms, and any messages exchanged. This is a significant responsibility — the admin's decision is final in this system — but because the evidence is comprehensive and timestamped, the decision is grounded in facts rather than personal impressions.

3.4.3 Authentication and Authorization

Authentication is handled by NextAuth.js. Users can sign up with email and password, or sign in with Google OAuth. NextAuth manages session tokens and provides the current user object to both server and client code via the session callback.

Role-based access control has three levels: RENTER (default), OWNER (same user, gains owner capabilities when they list an item), and ADMIN. In practice, RENTER and OWNER aren't separate roles — the same user can do both. The distinction is just that certain actions (like approving requests or signing return receipts as the owner) check whether you are the ownerId on the specific order.

ADMIN is a hard role assigned in the database. Admin users see the admin panel link in navigation and can access /admin/* routes. On the product detail page, admins see a read-only "Admin View" badge instead of buy/rent buttons — this prevents admins from accidentally creating orders while monitoring the platform.

Banned users (where isBanned = true in the database) are rejected at the NextAuth signIn callback before the session is created, so they can't log in at all. KYC verification is a gate specifically for listing items — an unverified user can browse and view items, but the server action for creating a listing checks verificationStatus === "APPROVED" and returns an error if not met.

One design decision worth explaining is how statusHistory is stored as a JSON field on RentalOrder rather than as a separate table. The status history is append-only and never queried independently — it's always read as part of the full order. Storing it as JSON avoids an extra join and keeps the order record self-contained. The field is a JSON array of objects like {"status": "APPROVED", "changedAt": "...", "note": "..."}.

A similar approach is used for returnPhotos on the rental order — storing the array of uploaded photo URLs directly on the order record rather than creating a separate evidence table. For a prototype this is fine; a production system might want a dedicated evidence table with more metadata per photo.

The walletBalance field on User is updated atomically using Prisma transactions. Every operation that touches a wallet uses prisma.$transaction([...]) to ensure that the debit and credit happen together or not at all. This prevents race conditions where two simultaneous requests could both read the same balance and both succeed, resulting in an overdraft.

3.4.6 PDF Receipt Generation

PDF generation was something I found harder than expected. My first idea was to use @react-pdf/renderer (a React-based PDF library), but getting it to work with Thai characters and custom fonts in a server-side Next.js environment turned out to be really complicated. I spent about two days on it before deciding to take a simpler approach.

The solution I ended up using is a print-optimized HTML page. The receipt page (app/rental/orders/[id]/receipt/page.tsx) is a server component that fetches the order data and renders a styled HTML document. It has a PrintBar client component at the top (hidden when printing) with a "Print / Save PDF" button, and an AutoPrint client component that triggers window.print() after a short delay when the page loads.

The print CSS uses @media print to hide the browser's native UI and the action bar, set margins to 20mm, and ensure the colors print correctly with -webkit-print-color-adjust: exact. When the user clicks the print button or opens the page, Chrome shows the native print dialog. They can choose "Save as PDF" and get a clean A4 document. It's not as automated as a true server-side PDF generation approach, but it produces a better-looking result for Thai text and requires no additional dependencies.

The two receipt types are: the Rental Contract (available at ?type=contract once the order is ACTIVE), which shows party names, item details, rental period, financial summary, agreed terms, and a signature block; and the Return Receipt (available at ?type=return after COMPLETED), which includes all of the above plus the return condition, damage assessment, final settlement amounts, and a return date.

Items have a soft-delete mechanism with a 24-hour grace period — when a seller marks an item for deletion, it stays visible on the marketplace for 24 hours with a warning banner (so active buyers aren't blindsided) and then disappears. Sellers can cancel the deletion within the grace period.

Performance

On a local development machine, page loads are fast — the homepage typically renders in under 500ms including the recommendation query. The recommendation algorithm runs a few database queries sequentially and then does in-memory scoring, which is fine for the expected scale. For a production deployment handling hundreds of simultaneous users, the recommendation query would need caching, but there's already a 5-minute cached wrapper in place for that.

4.2 Conclusion

This project went through five main phases, and each one taught me something different.

Phase 1: Requirement Analysis. The survey of PSU students confirmed the problem was real — most students had at some point needed equipment they didn't own, and most had had a bad experience with informal borrowing. The most common complaint was that borrowed items came back late (or not at all) with no recourse. This shaped the emphasis on the deposit escrow and the Digital Handshake evidence system. If I hadn't done the survey, I might have focused more on the discovery and listing features and less on the accountability mechanisms.

Phase 2: System Design. Designing the rental lifecycle state machine was the most intellectually challenging part of this phase. There are a lot of edge cases: what if the owner never responds? What if the item gets lost? What if one party confirms the handshake and the other doesn't? Working through all these scenarios before writing code saved a lot of debugging time later. I drew the state diagram probably four or five times before I was happy with it.

Phase 3: UI/UX Design and Frontend. I started with Figma wireframes for the main flows and built the frontend from there. The checkout wizard and the Digital Handshake screens were the most effort-intensive UI components. The checkout wizard needed to feel like a smooth multi-step flow rather than four separate pages, and the handshake needed to handle both photo upload and signature capture in one coherent experience. The signature canvas component took about two full days to get right, but I'm pretty happy with how it turned out.

Phase 4: Backend and Database. The server actions architecture of Next.js 14 made the backend cleaner than I expected. Writing the escrow settlement logic was tricky because I had to think carefully about what happens when multiple things go wrong at the same time — damage fees, late fees, and a potential dispute all in the same order. Getting the database transactions right (to avoid race conditions in wallet operations) was something I had to research and test carefully.

Phase 5: Testing and Iteration. The user testing session identified several UI issues I hadn't noticed during development, mostly around clarity of the status terminology (Thai terms for the rental states needed to be clearer) and the mobile experience for the signature canvas. Most of these were fixed before the final report, but a few are noted as known issues for Project II.

Overall, I think Project I achieved its main goal: demonstrating that a peer-to-peer rental system with proper accountability mechanisms is technically feasible within a university context. The core rental lifecycle works, the escrow system protects both parties, and the Digital Handshake creates the evidence record that makes dispute resolution possible. What's left for Project II is mostly making it production-ready — real payments, a mobile app, better performance, and proper deployment.

4.3 Problems and Obstacles

Honestly, there were quite a few moments where I got stuck. Here are the ones that cost the most time:

Understanding Prisma relations and JSON fields. Prisma is mostly intuitive, but I ran into a specific problem with JSON fields: when I stored the status history as a JSON-stringified array, Prisma returned it as a string, not a parsed array. This caused a runtime crash ("statusHistory.map is not a function") that took a while to track down. The fix was to always pass raw JavaScript arrays to Prisma for JSON fields and add a safe parse utility function for reading legacy rows.

Race conditions in wallet operations. Early on, I wrote wallet debit/credit as two separate database operations. During testing, I simulated two simultaneous approval actions and found that both could read the same wallet balance and both succeed — resulting in an overdraft. Wrapping everything in prisma.$transaction fixed it, but it required refactoring several server actions.

Canvas scaling for retina displays. The signature canvas looked blurry on high-DPI screens. The fix (scaling the canvas internal resolution by devicePixelRatio while keeping CSS dimensions at display size) is well documented online, but figuring out that this was the problem took longer than it should have because the blurriness was subtle on my development monitor.

PDF generation with Thai text. As described in Section 3.4.6, getting @react-pdf/renderer to handle Thai correctly in a Next.js server environment was too difficult for the time available. Switching to the browser print approach was the right call, but it cost about two days of experimentation first.

Zod v4 API changes. The project uses Zod for validation, and I upgraded to version 4 partway through development. Several API methods changed between v3 and v4 (notably .errors became .issues and the errorMap parameter syntax changed). This caused TypeScript errors that weren't immediately obvious because the types still seemed to match.

Time management. Exam weeks in October and November compressed the development schedule significantly. A few features that were planned for earlier ended up being pushed to the last few weeks. In hindsight, I should have started the backend work earlier and not assumed that the frontend would take as long as it did.

export async function createRentalOrder(data: CheckoutInput) {
  const session = await auth();

  const totalPaid = rentalFee + platformFee + securityDeposit;

  // Atomic: check balance AND debit in one transaction
  const [updatedUser, order] = await prisma.$transaction([
    prisma.user.update({
      where: { id: session.user.id, walletBalance: { gte: totalPaid } },
      data:  { walletBalance: { decrement: totalPaid } },
    }),
    prisma.rentalOrder.create({ data: { ...orderData } }),
  ]);

C.2 Rental Lifecycle Status Transition (Approval)

// lib/actions/rental-transitions.ts (excerpt)
export async function approveRentalRequest(orderId: string) {
  const session = await auth();
  const order   = await prisma.rentalOrder.findUnique({ where: { id: orderId } });

  if (order?.ownerId !== session?.user?.id) return { error: "Not authorized" };
  if (order.status !== "REQUESTED")         return { error: "Invalid status" };

  const newHistory = [
    ...(order.statusHistory as any[]),
    { status: "APPROVED", changedAt: new Date().toISOString(), note: "Owner approved" }
  ];

  await prisma.rentalOrder.update({
    where: { id: orderId },
    data:  { status: "APPROVED", statusHistory: newHistory },
  });

  // Notify renter
  await prisma.notification.create({
    data: { userId: order.renterId, type: "RENTAL_UPDATE",
            message: `เจ้าของอนุมัติคำขอเช่าของคุณแล้ว — ${order.refCode}`,
            link: `/rental/orders/${orderId}` }
  });
  return { success: true };
}

    data:  updateData,
  });

  if (updated.pickupRenterConfirm && updated.pickupOwnerConfirm) {
    await prisma.rentalOrder.update({
      where: { id: orderId },
      data:  { status: "ACTIVE", actualPickupAt: new Date(),
               statusHistory: [..., { status: "ACTIVE", changedAt: new Date() }] }
    });
  }
  return { success: true };
}
